# Feature Specification: Othello 対 CPU 対戦 MVP

**Feature Branch**: `001-othello-cpu-match`

**Created**: 2026-05-30

**Status**: Draft

**Input**: User description: "モダンで直感的な UI を持つオセロ（リバーシ）対戦アプリの v1。8×8 盤、公式ルール準拠、人間 vs CPU、先手/後手の選択、CPU の強さ 3 段階（弱・普通・強）、モダンでミニマルなデザイン、盤面・石のアニメーション、スマホ/PC 両対応、チュートリアル無しで遊べる直感性。オンライン/ローカル 2 人対戦・ダークモード等のテーマ設定は今後の検討事項。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 既定設定でオセロを 1 局通して遊ぶ (Priority: P1)

オセロのルールを知っているユーザーがアプリを開き、説明を読まずに「対戦開始」を押すだけで、既定の設定（先手＝プレイヤー、CPU 強さ＝普通）で 1 局を最後まで遊び、勝敗を確認できる。

**Why this priority**: これが本プロダクトの中核体験であり、これ単体でもアプリとして成立する MVP。手番制御・合法手判定・石の反転・パス・終局判定・勝敗表示という公式ルールの全要素が一通り動かないと「オセロアプリ」と呼べないため、まずこの動線が完成することが最重要。

**Independent Test**: アプリを開いて「対戦開始」を押す → プレイヤーが石を置く → CPU が応手する → 交互に進めて終局する、までを手動で 1 局通して実行できれば独立に検証可能。盤面初期配置・反転・パス・終局・勝敗表示の各挙動が公式ルール通りであることをこの 1 シナリオで確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーがアプリのトップ画面にいる, **When** 「対戦開始」相当の単一アクションを実行する, **Then** 中央 4 マスに公式の初期配置（黒2・白2 を斜めに）で並んだ 8×8 盤と、自分の手番表示と、現在の合法手の視覚的ヒントが表示される
2. **Given** 自分の手番で合法手が存在する, **When** 合法手のマスを選択する, **Then** そのマスに自石が置かれ、挟まれた相手石が全方向で反転し、手番が CPU に移る
3. **Given** 自分の手番だが合法手が 0 個である, **When** （ユーザーの操作なしに）一定時間が経過する, **Then** 「パス」が明示的に通知され、手番が相手に移る
4. **Given** 両者連続パスまたは盤面が満杯になった, **When** 終局条件が満たされる, **Then** 最終スコア（黒石数 - 白石数）と勝敗（勝ち／負け／引き分け）が表示され、もう一度遊ぶ導線が提示される
5. **Given** 自分の手番で非合法マスを選択する, **When** 操作が行われる, **Then** 石は置かれず、なぜ非合法かを暗示する視覚フィードバック（合法手の強調等）が返り、手番は維持される

---

### User Story 2 - 先手・後手を選んで対戦する (Priority: P2)

ユーザーは対戦開始前に「自分が先手（黒）」か「自分が後手（白）」かを選択でき、選択した側でゲームを開始できる。

**Why this priority**: オセロでは先手後手の選好が学習・楽しさに直結するが、P1 の既定動線（プレイヤー＝先手＝黒）で 1 局成立するため、独立した拡張として後回し可能。とはいえ「対戦相手の固定化を避ける」ことは初回体験の幅を大きく広げるため P2。

**Independent Test**: 対戦開始前のセットアップ画面で「後手（白）」を選んで開始 → CPU が黒で先に着手する → ユーザーが白として応手する、までを 1 局実行できれば独立に検証可能。

**Acceptance Scenarios**:

1. **Given** トップ／セットアップ画面で先手後手の選択 UI が表示されている, **When** ユーザーが「後手（白）」を選び対戦開始する, **Then** 盤が初期配置で表示され、最初の手番は CPU（黒）に割り当てられ、CPU が思考表示の後に着手する
2. **Given** ユーザーが「先手（黒）」を選んで対戦開始する, **When** 盤が表示される, **Then** 最初の手番はプレイヤー（黒）で、合法手ヒントが表示される
3. **Given** ユーザーが先手後手の選択をしないまま対戦開始した, **When** ゲームが始まる, **Then** 既定値（先手＝プレイヤー＝黒）が適用され、選択 UI が次回起動時には記憶されている

---

### User Story 3 - CPU の強さを 3 段階から選んで対戦する (Priority: P2)

ユーザーは対戦開始前に CPU の強さを「弱い」「普通」「強い」の 3 段階から選択でき、選んだ強さに応じて CPU の応手の質が体感できるレベルで変化する。

**Why this priority**: 初心者から上級者まで楽しめるための重要要素であり、リプレイ性に直結する。一方で P1（既定＝普通で 1 局遊べる）が成立すれば最低限のプレイ体験は提供できるため P2。先手後手選択（Story 2）と同列の独立な拡張。

**Independent Test**: セットアップ画面で「弱い」を選び対戦 → 別途「強い」を選び対戦、の 2 局を比較し、強さ設定により応手の傾向（例: 角を取る／取らせる頻度、悪手率）が異なることを確認できれば独立に検証可能。

**Acceptance Scenarios**:

1. **Given** セットアップ画面で強さ選択 UI が表示されている, **When** ユーザーが「弱い」を選んで対戦開始する, **Then** CPU は「弱い」ロジックで応手し、思考時間も「強い」より短いか同等になる
2. **Given** 「強い」設定で対戦中, **When** プレイヤーが角を取れる手を見送る局面が複数回発生する, **Then** 「強い」CPU は同じ局面の少なくとも一部で角や辺の好手を選び、「弱い」CPU よりも有意に高い勝率を出す（後述 SC-005 参照）
3. **Given** ユーザーが強さを選ばずに対戦開始した, **When** ゲームが始まる, **Then** 既定値「普通」が適用される

---

### User Story 4 - スマートフォンと PC の両方で快適に操作する (Priority: P2)

同じアプリを PC のブラウザでもスマートフォンのブラウザでも開いて、いずれの環境でも盤面全体が一画面に収まり、タップ／クリックで石を置けて遊び切れる。

**Why this priority**: ユーザーがどのデバイスから来るかは制御できず、片方しか使えないと「直感的に遊べる」という目標が崩れる。P1 の機能集合で動くため別ストーリーとして独立。

**Independent Test**: 同じ URL を PC ブラウザと一般的なスマートフォン縦持ち画面（幅 360–430px 想定）でそれぞれ開き、両方で盤面が縦横スクロールなしに全マス見えてタップで着手できれば独立に検証可能。

**Acceptance Scenarios**:

1. **Given** 幅 360px のスマートフォン縦持ち画面でアプリを開く, **When** 対戦画面が表示される, **Then** 8×8 盤と現手番表示・スコア表示が縦スクロールなしで視界に収まり、各マスは指でタップしやすい大きさになる
2. **Given** 幅 1280px 以上の PC 画面でアプリを開く, **When** 対戦画面が表示される, **Then** 盤がウィンドウ中央に適切なサイズで表示され、補助情報（スコア・手番・操作）が盤の周辺にレイアウトされる
3. **Given** 画面を縦持ち→横持ちに回転する, **When** 回転が完了する, **Then** 盤面状態は失われず、新しい画面サイズに合わせてレイアウトが再構成される

---

### User Story 5 - 視覚フィードバックで状況が直感的に分かる (Priority: P3)

合法手の候補、石の配置、反転、手番の切り替わり、CPU の思考中、勝敗の確定などの主要な状態変化が、テキスト説明に頼らずアニメーション／色／配置の変化だけで判別できる。

**Why this priority**: 「チュートリアル無しで遊べる」という目標の達成度を実質的に決定づける品質要素。P1 のロジックが動けば最低限プレイ可能だが、フィードバックが弱いとユーザーが「自分の手が通ったのか」「次は誰の番か」を見失う。P1 が成立した後に磨き込むべき層なので P3。

**Independent Test**: ルールを知らない協力者にアプリを渡し、口頭の説明なしで盤面・スコア・手番・終局を理解しながら 1 局を遊び切ってもらえれば独立に検証可能。

**Acceptance Scenarios**:

1. **Given** 自分の手番である, **When** 盤面が描画される, **Then** 合法手のマスはそれ以外と視覚的に明確に区別されて表示される
2. **Given** 石を置いた, **When** 反転処理が起きる, **Then** 反転される全石が短時間のアニメーションで色が裏返り、ユーザーは「どの石が自分のものになったか」をテキストなしで把握できる
3. **Given** CPU の手番に入った, **When** CPU が思考中である, **Then** CPU 側であることと「今応手を考えている」ことを示す視覚的インジケータが表示される
4. **Given** 終局した, **When** 結果画面に遷移する, **Then** 勝敗が色・配置・スコア表示で一目で分かる形になる

---

### Edge Cases

- **片側のみ合法手が無い局面**: 一方のプレイヤーに合法手が無く、他方には合法手がある場合、自動でパスを通知し、合法手のあるプレイヤーの番に進む。プレイヤーがパスを操作する必要は無い。
- **両者連続パス**: 連続してパスが発生したら盤が満杯でなくても終局とする。
- **盤が満杯になる前にどちらかが全滅**: 一方の色の石が 0 個になったら終局とし、残った色を勝者とする。
- **非合法マスの選択**: 何も置かれず、合法手の強調や軽い触覚的・視覚的なはじき返しで応じる。エラーモーダルは出さない。
- **CPU 応手中のプレイヤー操作**: プレイヤーの手番ではない間の盤面タップは無視され、誤って手が進むことはない。
- **対戦途中のリロード／タブ切替**: 進行中の対局を再開できる扱いとするか破棄するかは v1 ではセッション内保持のみとし、リロードで破棄される（学習用途のため複雑な永続化は持たない）。
- **画面回転中**: 盤面の論理状態は維持し、再レイアウトのみ行う。
- **アクセシビリティ補助操作**: マウス／タッチに加え、キーボード操作（タブ・矢印・Enter 相当）でも合法手を選んで着手できる（直感性の補完として P3 アクセプタンス層に含む）。

## Requirements *(mandatory)*

### Functional Requirements

**ルール準拠**

- **FR-001**: システムは盤面を 8×8 で初期化し、中央 4 マスに公式の初期配置（D4 白・E5 白・D5 黒・E4 黒、もしくは等価な向き）で 4 石を配置 MUST する
- **FR-002**: システムは手番プレイヤーの「合法手」を、その色の石を置くことで少なくとも 1 方向で相手石を挟める空きマス全集合として算出 MUST する
- **FR-003**: システムは合法手にのみ着手を許可 MUST し、非合法マスでの着手要求は受理してはならない (MUST NOT)
- **FR-004**: システムは着手によって挟まれた相手石を 8 方向すべてについて反転 MUST する
- **FR-005**: システムは合法手が無いプレイヤーの手番を自動的にパス扱いとし、相手に手番を移譲 MUST する
- **FR-006**: システムは両者が連続パス、もしくはどちらかの石が 0 になる、もしくは盤面が満杯になった時点で対局を終局 MUST する
- **FR-007**: システムは終局時に各色の石数を集計し、多い方を勝者、同数なら引き分けとして表示 MUST する

**ゲームセットアップ**

- **FR-008**: システムはプレイヤーに対戦開始前に「先手（黒）」「後手（白）」を選択させる手段を提供 MUST する
- **FR-009**: システムはプレイヤーに対戦開始前に CPU の強さを「弱い」「普通」「強い」の 3 段階から選択させる手段を提供 MUST する
- **FR-010**: システムはプレイヤーが選択をせずに対戦を開始した場合、既定値（先手＝プレイヤー（黒）、強さ＝普通）を適用 MUST する
- **FR-011**: システムはプレイヤーが直前に使った先手後手と強さの設定を次回のセットアップ画面で初期選択として再現 SHOULD する

**CPU 挙動**

- **FR-012**: CPU は自身の手番で必ず合法手集合から 1 手を選んで着手 MUST し、合法手が無い場合のみパス MUST する
- **FR-013**: 「弱い」CPU は判断材料を最小限に抑え、合法手の中から相対的に簡素な基準で選ぶ（例: ランダム選択や近視眼的評価）。具体アルゴリズムは plan/実装段階で決定する。
- **FR-014**: 「普通」CPU は「弱い」より明確に強く、「強い」より明確に弱い水準で応手 MUST する（後述 SC-005 で計測）。
- **FR-015**: 「強い」CPU は中盤以降に角・辺の重要度を意識した手を選び、対局終盤では先読みに基づいて応手 SHOULD する。具体アルゴリズム・探索深度は plan/実装段階で決定する。
- **FR-016**: CPU の応手は強さに関わらず 1 手あたり一定時間以内に返る（SC-003 参照）こと MUST。

**UI / UX**

- **FR-017**: システムは自分の手番中、合法手のマスを非合法マスと視覚的に区別して表示 MUST する
- **FR-018**: システムは石を置く操作と石の反転を、開始から完了まで合計 300ms 以下のアニメーションで視覚化 SHOULD する（憲法 II 準拠）
- **FR-019**: システムは現在の手番、両者の現スコア（石数）、CPU の思考中状態を常に表示 MUST する
- **FR-020**: システムは PC（幅 1024px 以上）と一般的なスマートフォン縦持ち（幅 360px 以上）両方で、盤と主要な情報領域がスクロール無しで一画面に収まるレイアウト MUST を提供する
- **FR-021**: システムはタッチ・マウスクリック・キーボード操作（合法手のフォーカス移動と決定）で着手可能 SHOULD であり、説明文を読まなくても操作方法が発見可能 MUST である
- **FR-022**: システムは対戦中、相手の手番ではプレイヤーの誤クリックを無視 MUST する

**対局終了とリスタート**

- **FR-023**: システムは終局時に最終スコアと勝敗を表示し、「もう一度遊ぶ」相当の動線を提供 MUST する
- **FR-024**: 「もう一度遊ぶ」操作はセットアップ画面に戻り、直前の先手後手・強さ設定を初期値として再選択可能にする SHOULD

### Key Entities

- **Board（盤面）**: 8×8 マスの集合。各マスは「空」「黒石」「白石」のいずれか。
- **Move（着手）**: 「どの色がどのマスに石を置いたか」と、その結果反転される石の座標リスト。
- **Game（対局）**: 初期化された盤面、現在の手番、両者の色割当、強さ設定、手順履歴、現在の状態（進行中／終局／引き分け）を持つ単位。
- **PlayerProfile（プレイヤー設定）**: 「プレイヤー側の色（黒/白）」「CPU の強さレベル」など、ゲーム開始前に決定される選好。次回再現の対象。
- **CPUStrategy（CPU 戦略）**: 強さレベルごとに紐づく応手選択ロジックのアイデンティティ（実装は分離）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001（ルール正しさ）**: 標準オセロのルール検証シナリオを 100% 通過する（初期配置、8 方向反転、合法手判定、パス、終局、勝敗判定）。1 件も不一致が無いこと。
- **SC-002（初手到達時間）**: アプリ初回読み込みから「自分の最初の着手が可能になる」までが、平均的なネットワーク・端末で 5 秒以内。
- **SC-003（CPU 応答性）**: いずれの強さ設定でも、CPU の 1 手あたり応答時間が中央値 1 秒以内・95 パーセンタイル 3 秒以内。
- **SC-004（直感性）**: ルール既知のユーザーが、説明文を読まずにアプリを開いてから 1 局終局までを 90% 以上の確率で完了できる（ユーザビリティテスト 10 人規模の合格率で評価）。
- **SC-005（強さ差別化）**: 同条件 100 局自己対戦で、「強い CPU」が「弱い CPU」に対して 70% 以上、「強い CPU」が「普通 CPU」に対して 55% 以上の勝率を出す。
- **SC-006（モバイル可読性）**: 幅 360px の縦持ちビューポートで、対局画面の盤と主要情報が縦横ともにスクロール無しで全表示される（測定: 各マスの最小タップ領域が 32px 以上を満たす）。
- **SC-007（リプレイ率）**: 対局終了画面から「もう一度遊ぶ」が選ばれる割合が 40% 以上（v1 リリース後 1 か月の計測目標）。
- **SC-008（アニメーション体感）**: 着手から反転完了までのアニメーションが 300ms 以下に収まり、ユーザーが「処理が引っかかった」と知覚しない（憲法 II 準拠）。

## Assumptions

- **対局形態の範囲**: v1 はプレイヤー 1 名 vs CPU 1 体のみを対象とする。ユーザーが「今後詰めたい点」として挙げた「オンライン対戦」「ローカル 2 人対戦」は v1 スコープ外とし、後続のフィーチャ仕様で別途定義する。
- **デザインテーマ**: v1 ではミニマルかつ汎用的な単一テーマ（コントラスト確保・モダンな配色）で提供し、ダークモード切替や複数テーマは v1 スコープ外とする。
- **アカウント・永続化**: v1 ではユーザーアカウントを持たず、対局結果や設定の保存はブラウザのローカル保存（端末内）に限定する。サーバ側の永続化や同期は行わない。
- **オフライン動作**: 初回ロード後は基本的にローカルで完結する（CPU は端末で動く）。サーバ側の常時オンラインは前提としない。
- **対応ブラウザ**: 直近の主要ブラウザ（Chrome / Safari / Edge / Firefox の最新メジャー 2 バージョン）を想定する。
- **アクセシビリティ水準**: 憲法 III に従い、キーボード操作とスクリーンリーダーでの主要機能利用を「SHOULD」レベルで満たす。完全な WCAG 2.1 AA 適合の認証取得は v1 範囲外。
- **計測**: SC-007 等の利用率系指標は「リリース後にプロダクト解析で測定可能になる」前提。v1 で計装の詳細は plan 段階で定義する。
- **言語**: v1 の UI 文言は日本語のみ。多言語対応は v1 スコープ外。
- **対局リプレイ・棋譜エクスポート・対局時計**: v1 スコープ外。
